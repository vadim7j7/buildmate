# LANGUAGE Style Guide

Code style conventions for FRAMEWORK development.

## 1. File Organization

### Import Order

Organize imports in this order (with blank lines between groups):

1. Standard library imports
2. Third-party imports
3. Framework imports
4. Local/project imports

```LANGUAGE
# Standard library
import os
from datetime import datetime

# Third-party
import library

# Framework
from framework import Module

# Local
from app.services import MyService
```

## 2. Naming Conventions

| Type | Convention | Example |
|------|------------|---------|
| Files | snake_case | `user_service.py` |
| Classes | PascalCase | `UserService` |
| Functions | snake_case | `get_user_by_id` |
| Variables | snake_case | `user_count` |
| Constants | UPPER_SNAKE | `MAX_RETRIES` |
| Private | _prefix | `_internal_method` |

## 3. Type Annotations

**Always use type annotations** for function parameters and return types.

```LANGUAGE
# Good
def get_user(user_id: int) -> User | None:
    ...

# Bad
def get_user(user_id):
    ...
```

### Union Types

Use modern union syntax:

```LANGUAGE
# Good
name: str | None = None

# Avoid (older syntax)
name: Optional[str] = None
```

## 4. Documentation

### Docstrings

Use DOCSTRING_STYLE docstrings for public functions and classes:

```LANGUAGE
def create_user(name: str, email: str) -> User:
    """
    Create a new user.

    Args:
        name: The user's display name.
        email: The user's email address.

    Returns:
        The created User object.

    Raises:
        ValidationError: If email is invalid.
    """
```

### Comments

- Use comments sparingly
- Explain "why", not "what"
- Keep comments up to date

## 5. Code Formatting

### Line Length

Maximum LINE_LENGTH characters per line.

### Indentation

Use INDENT_SIZE spaces for indentation.

### Blank Lines

- 2 blank lines between top-level definitions
- 1 blank line between methods in a class
- Use blank lines sparingly within functions

## 6. Error Handling

### Exceptions

```LANGUAGE
# Good - specific exceptions
try:
    result = risky_operation()
except SpecificError as e:
    logger.error(f"Operation failed: {e}")
    raise

# Bad - bare except
try:
    result = risky_operation()
except:
    pass
```

### Error Messages

- Be descriptive
- Include context
- Don't expose internals to users

## 8. Testing Conventions

### Test File Naming

- `test_<module>.py` or `<module>_test.py`
- Match the structure of source files

### Test Function Naming

```LANGUAGE
def test_<function>_<scenario>_<expected>():
    # test_create_user_with_valid_data_succeeds
    # test_create_user_with_empty_name_raises_error
```

## 9. Linting and Formatting

### Required Tools

- Linter: LINTER_TOOL
- Formatter: FORMATTER_TOOL
- Type checker: TYPE_CHECKER (if applicable)

### Commands

```bash
# Lint
LINT_COMMAND

# Format
FORMAT_COMMAND

# Type check
TYPECHECK_COMMAND
```

## 10. Git Conventions

### Commit Messages

```
type(scope): description

- feat: New feature
- fix: Bug fix
- docs: Documentation
- refactor: Code refactoring
- test: Tests
- chore: Maintenance
```

### Branch Naming

```
feature/<description>
fix/<description>
refactor/<description>
```
